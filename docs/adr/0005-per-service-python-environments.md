# Per-service Python environments for model-serving subprocesses

**Status:** accepted

While spiking the STT server (Milestone 1.3), installing `nemo_toolkit[asr]` into a fresh Python 3.12 environment on dgx1 failed: its dependency resolver backtracked to `numba==0.53.1` (a 2021-era release requiring Python <3.10) to satisfy a conflict between `numba`'s numpy pin and another package demanding `numpy==2.5.1`. Pinning `numpy<2` alongside `nemo_toolkit[asr]` resolves cleanly (numba 0.66.0, llvmlite 0.48.0). Separately, `vllm` resolves fine against modern numpy on its own. Forcing both into one shared environment risks exactly this kind of cross-package constraint conflict, and there's no reason to share an environment at all: each model server (vLLM, STT, TTS, VLM) already runs as its own OS subprocess per ADR 0002, so each also gets its own dedicated venv with its own pinned dependencies, installed at deploy time. The `hugo` package's own base dependencies (typer, pydantic, websockets, etc.) stay lightweight and are installed into every service venv alongside that service's heavy ML dependencies, so the shared server/client code in `src/hugo/servers/` and `src/hugo/voice/` remains importable everywhere without forcing one global lockfile to satisfy every model library's constraints simultaneously.
